blockr.bi renderers — what we're building

Tables, charts and tiles are pure renderers. They draw a rectangular data frame — usually tidy/long, sometimes deliberately wide (a pivot grid, a correlation matrix) — and nothing more. Every summarize, pivot_wider, correlate happens upstream, as ordinary composable blocks.

Design mockup · 2026-06-13 · static, throwaway — for deciding the target before any code

A renderer maps columns → roles. It never aggregates, reshapes, or computes.

If you can see a number in the output, that number was already in the frame the renderer received. The block UI is just a short list of column pickers.

01The pipeline is the product

The same three or four upstream verbs feed every renderer. Reading a dashboard pipeline tells you exactly what it does — no hidden compute inside the renderer.

dataset→ summarize→ ◰ tile KPI cards — one number per measure
dataset→ group_by + summarize→ pivot_wider→ ▦ table crosstab — rows × arms
dataset→ correlate→ ▦ table heatmap = table with cell_color
dataset→ summarize→ ▦ chart bar — group + metric + agg
The one allowed exception: a chart self-aggregates, because the visual encoding is the aggregation (a bar = sum-of-group). Tables and tiles take an already-shaped frame.

02Tile — one value renderer; the body follows what you map

A tile shows a value. Map a target → it grows a delta; map a max → it grows a progress fill; map a by column → one tile per category. No "showcase zoo" and no recipe templates — the body is a function of the roles you map. (Spark descoped — see below.)

A sparkline tile pays a tax nothing else in the tile does: it needs a spark_x ordering role and a raw-row passthrough (stat = identity → a list-column of vectors per cell) and a charting dependency (echarts) just to draw a line. Three kinds of complexity for one body style — and it's the reason the shaping pipeline grows list-columns at all.

A sparkline is a tiny chart — that's the chart renderer's job, not the tile's. Want a trend in a card? Drop a compact line chart in. That keeps the tile a pure scalar renderer. Revisit later as "chart-in-a-card" only if it earns its keep.

02bSame numbers, two layouts — cards or a colored scorecard

A tile's job is to show a few important numbers boldly. Sometimes that's free-floating cards; sometimes a compact, colored, structured grid — a scorecard. Both are the tile (static, bold, a handful of numbers) — the opposite of a chart, and distinct from the data table in §03 (dense, scrollable, sortable, drillable). layout is one small toggle: cards · table.

One renderer — bold numbers, free or structured. Color (value/status tint) is automatic & purposeful; the old recipe templates, spark, and the elaborate color-by/intensity config are gone.
The line vs the data table (§03): tile-table = static, bold, a handful of numbers, decorative color (a scorecard). Data table = dense rows, scan/sort/search, drill, sections, composite .fmt cells. Same HTML-table shape, different job. If that line blurs in practice, the tile-table is the thing at risk of overlapping — worth watching.

02cOne element set — cards and tables draw the same parts

Cards and the table layout are not two feature sets. They render the same elements; the layout only decides placement. A card stacks them; a table hoists the shared ones (title, unit) into headers and packs value + delta + fill + color into cells.

the elements

1  title — what the number is 2  value — the number 3  unit — %, $, cm … 4  max — reference → a fill 5  color — tint by value / status 6  delta — trend vs a reference (▲/▼)

Everything is present in both modes — nothing lives in one that can't in the other. The card stacks the parts; the table puts title + unit in the column header and group in the row header, leaving value + delta + fill + color in the cell.

Multi-column table — measures across, groups down. Each column is one value measure; each row is a by level. Every cell is still a mini-tile with the same elements; the shared ones are hoisted — unit & measure title → column header, group → row header. It reads as a table (aligned rows, hairline rules), not a card grid.

“More than one value — do we lose the trend?” No. The delta (▲/▼ vs a reference) is a per-cell element — every cell keeps its own, however many measures. Revenue & Conversion below each carry value + delta. The only thing descoped is the literal sparkline (a line over time) — that's the chart's job. max is per-cell too — and it stacks with delta: a column with a max (Budget) shows a fill and keeps its delta, since the two answer different questions (delta = vs last period; fill = vs target).

Same elements, a different mix per column, and it reads as a table, not a card grid. The cards view of this same frame is one card per cell (Region × measure), with the identical elements.

The data contract is unchanged: still long/tidy group · measure · value (+ optional max), exactly like everything upstream. The matrix is produced by the renderer hoisting measure into columns at draw time — a presentation of the tidy frame, not a pivot_wider baked into the data.

03Table — one renderer for every table

Table 1, a regression table, a survival table, a crosstab — all the same display: rows × an optional spread dimension × composite cells. Fed by tidy shapers; structure rides in hidden dotted columns (.section, .indent, .fmt).
