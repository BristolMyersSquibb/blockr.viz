KPI tile slot models — what others do

The question behind "is delta the right fixed element?": should a tile expose fixed semantic fields (value, delta, target) or open generic slots (supertext / main / subtext you fill with anything)? Surveyed 20+ design systems, component libraries and BI tools.

Research 2026-06-13 · compiled from 3 source sweeps (web design systems · BI platforms · R value-boxes + UX anatomy guides) · companion to the tile mockup

The market's center of gravity is a hybrid: a typed value core (which is what buys auto-formatting, auto trend color, and target math) + free slots (overline / caption) for anything else.

Pure-generic (your super/main/sub idea) has real precedent — bslib value_box, shinydashboard infoBox, Grafana text mode — but it gives up number formatting, trend color, and target math. Pure-fixed (Ant Statistic) strains: it had to bolt the delta onto prefix because it has no delta field.

01Three camps — and where the field actually sits

Every product lands in one of three places. The largest cluster, and where the newest "metric" surfaces (Tableau Pulse 2024, Looker Studio Scorecard) are converging, is the hybrid.

Generic slots

You assemble a card; the system gives no metric vocabulary.

  • bslib · title/value/.../showcase
  • shinydashboard · infoBox(title,value,subtitle)
  • flexdashboard · value/caption
  • shadcn/ui · Card* by convention
  • Power BI · Card / multi-row / new Card
  • Tableau canvas · measures → marks
  • Grafana Stat · text mode + thresholds
  • Polaris · Material 3 · Geist

+ max layout freedom
− no format, no trend color, no target math — you re-do it per card

Fixed semantic

You pass data; the component formats & styles it. Rigid.

  • Ant Design · Statistic: title/value/prefix/suffix/precision
  • Carbon · BigNumbers, Gauge, Meter (status ranges)

+ smart defaults (precision, separators, formatter)
− no delta field → Ant bolts ▲ into prefix + manual valueStyle color. Strains when a metric needs something unforeseen.

Hybrid center of gravity

Named value core with smarts, + free slots around it.

  • Chakra · Label/ValueText/HelpText/Up·DownIndicator
  • Tremor · Metric + auto-colored BadgeDelta + ProgressBar
  • Looker Studio · metric + comparison-mode + target + sparkline
  • Tableau Pulse · Measure/Comparison/Goal
  • Palantir · typed value+format+rules, generic 2nd metric
  • Sisense · Qlik · Metabase Trend · Mode · ServiceNow

+ smart where structure pays off (format, trend color, target)
+ generic where flexibility matters (caption, overline, embedded spark)

03Five patterns worth stealing

PatternWhat it is & who does it
Polarity flag Up isn't always good (churn, cost, incidents). A typed delta needs a "good when ↑/↓" flag, or green/red lies. Tremor isIncreasePositive · Looker "Positive Values Are Bad" · Power BI KPI Direction · Metabase reversible.
Comparison mode Don't hardwire "delta". Pick a mode: prior-period (auto-derived from a date breakout) or explicit value/metric target. Looker Studio Comparison type · Tableau Pulse Comparison · Metabase "Add comparison" · Mode Indicator.
Sparkline XOR progress A cell shows a trend line or a progress-to-target — not both; one UX switch. Looker Studio makes them mutually exclusive. Clean constraint to copy.
Spark = separate primitive Almost nobody bakes the sparkline into the stat schema — it's a chart dropped into the tile. Tremor Spark*, MUI SparkLineChart, Stripe Sparkline. Power BI/Tableau: only in tables. Validates our "spark = the chart's job" descope.
Many metrics → a table Every tool's dense answer is a metric / highlight table (rows = entities, cols = measures, conditional color), not an endless card grid. Tableau highlight tables · Power BI matrix + sparklines · Metabase color-range. Validates our layout = table.

04What this means for the tile block

Answer to "keep delta open?": go hybrid, leaning semantic. Type the few things only structure can make smart; leave a generic escape hatch for the rest. This is also exactly the codebase's "small basis set of deep blocks" philosophy — deep on format/trend/target, generic on captions.

Typed core — earns its smarts

  • value + format → commas / % / $ / compact, automatic
  • delta (or value−compare) → auto ▲▼ + green/red, + polarity flag
  • max / target → fill + "% of target" + gap math
  • unit → suffix/prefix
  • color → value/status tint (auto)

Open slots — fill with anything

  • overline — your "supertext" (date period, category, source)
  • caption — your "subtext" (footnote, context, free string)
  • (later) showcase — an icon or embedded mini-chart, bslib-style

These are the "fill it with whatever" slots you wanted — kept, but alongside the typed core, not instead of it.

Why not pure-generic (your super/main/sub idea alone): you'd lose the three things that make a tile feel finished — number formatting, automatic trend color, and target/progress math — and re-implement them by hand on every tile. bslib/shinydashboard sit there and pay exactly that tax.
Why not pure-fixed: Ant Statistic has no delta field, so the canonical "▲ 11.28%" is hand-built in prefix + valueStyle. A rigid schema breaks the moment a metric needs a part the designers didn't foresee — hence the free overline/caption.

Mapped onto our mockup's element set: title · value · unit · delta · max(→fill) · color stay the typed core; we add overline + caption as the open slots. Same two layouts (cards · table) still apply — both just render whichever of these are present.
